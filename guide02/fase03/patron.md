## Patrón del diseño de software (IDE de Desarrollo)

En la capa de presentación se implementará el enfoque MVC utilizando Java Swing para construir una interfaz gráfica dinámica e intuitiva que permita registrar productos, gestionar ventas y visualizar reportes mediante componentes como JTable, JChart y otros elementos gráficos de Swing. La capa de aplicación concentrará las reglas del negocio, incluyendo el cálculo de IGV, el control de stock y la generación de boletas, mientras que la persistencia de datos será manejada a través de arrays en memoria. Estos arrays se gestionarán directamente desde los servicios correspondientes, como ProductoService, VentaService y ReporteService, lo que elimina la necesidad de una base de datos externa. Todo el sistema se desarrollará en Java SE 17 LTS, aprovechando su robustez y soporte a largo plazo. Se empleará el IDE Apache NetBeans 17 por su integración con Java Swing y su entorno visual amigable.
## Patrón de la arquitectura del software

Para este prototipo se aplicó el patrón de 2 capas, complementado con el enfoque MVC en la capa de presentación. La capa de presentación gestiona la interacción con el usuario mediante vistas web o móviles, controladores REST y modelos de datos (DTO), permitiendo registrar productos, ventas y servicios de forma intuitiva. En este modelo, los datos se gestionan en arrays en memoria en lugar de depender de una base de datos, lo que simplifica el diseño y mejora la velocidad de procesamiento, pero limita la persistencia de los datos entre sesiones.
La capa de aplicación o dominio centraliza las reglas del negocio: control de stock, cálculo de IGV, margen de ganancia en servicios, registro de compras y ventas, así como la generación de boletas y reportes periódicos. En esta capa, los datos se almacenan temporalmente en arrays, lo que asegura un acceso rápido y eficiente a la información sin la necesidad de realizar operaciones de lectura/escritura en una base de datos. Aunque este enfoque facilita la implementación y mejora el rendimiento durante la ejecución, los datos no se conservan entre sesiones del sistema. Además, la estructura modular hace más sencilla la incorporación de nuevas funcionalidades, como reportes personalizados o validaciones automáticas de boletas, siempre que no sea necesario persistir la información más allá de la ejecución en curso.
